# PUBG Buddy — 产品需求文档(PRD)

> 工作代号 `pubg-buddy`,可随时改名。
> 版本:v0.1 · 日期:2026-07-17 · 状态:待开发

## 1. 产品概述

**一句话定位**:个人自用的 PUBG 桌面助手——桌面 App 查战绩、赛后自动出报告、进局自动看队友、游戏内地图标记刷车点/密室。

**核心价值**(相对 pubg.op.gg 等现成网站):

1. **自动**:赛后无操作自动弹报告;进局自动拉队友战绩(Overwolf roster)
2. **积累**:比赛与 telemetry 数据永久存本地,能做网站做不了的纵向分析(跳点复盘、死亡画像)
3. **游戏内**:浮窗与地图标记直接叠在游戏上,不用切屏

**明确不做**(Non-goals):

- 登录、多用户、权限体系——本地单人使用,零账号
- 云端部署、公网服务——一切数据在本机
- 任何实时敌情信息(读内存/抓包)——违反 ToS,红线
- 复杂的安全加固——API Key 存本地配置即可

## 2. 用户与场景

用户就是开发者本人 + 固定车队朋友(后期发安装包)。

| 场景 | 现状痛点 | 本产品体验 |
|---|---|---|
| 组到随机队友,想知道是大腿还是人机 | 切屏开网站、手动输名字 | 进局后浮窗自动列出全部队友的 K/D、吃鸡率 |
| 打完一局想复盘 | 网站数据延迟、信息散 | 赛后 1~2 分钟自动弹本局报告,全员数据+队友高亮 |
| 记不住刷车点/密室位置 | 第二屏开攻略图对照 | 游戏内按键呼出标记地图;完全体直接贴合 M 地图 |
| 想知道自己为什么上不了分 | 游戏内统计太简陋 | 死亡画像、跳点胜率、对枪命中率纵向曲线 |

## 3. 能力边界(所有需求的前提)

来自 PUBG 官方 API 的硬约束,产品设计必须服从:

| 约束 | 影响 |
|---|---|
| 无实时接口,比赛数据赛后几分钟才可查 | 「进局看全场 100 人」做不到;队友信息走 Overwolf roster |
| 默认限流 10 请求/分钟(429) | 必须缓存;批量接口(一次 10 人)优先 |
| `/matches` 与 telemetry **不计限流** | 比赛详情、telemetry 分析随便拉,数据底座免费 |
| 比赛数据不可变 | 永久缓存入库,积累历史库 |
| 名字查询要求精确匹配 | 队友名字来源必须精确(GEP roster 而非 OCR) |

合规边界:仅使用官方 API + Overwolf 白名单平台 + 屏幕捕获(与 OBS 同级),不注入游戏、不读内存、不抓包。

## 4. 功能需求(按阶段)

### P0 · 阶段一:数据底座 + 桌面 App

| # | 功能 | 说明 | 验收标准 |
|---|---|---|---|
| F1 | 设置页 | 填 API Key、绑定自己游戏名、平台(steam/kakao)、轮询开关 | Key 校验通过即可用;配置持久化 |
| F2 | 玩家查询 | 输名字查任意玩家:赛季/生涯战绩,分 solo/duo/squad × FPP/TPP | 缓存命中 <300ms;未命中 <2s |
| F3 | 我的仪表盘 | 本赛季概览卡片(K/D、场均伤害、吃鸡率、Top10 率)+ 近期比赛列表 | 打开 App 即见,无需操作 |
| F4 | 比赛详情页 | 本局全部玩家:排名、击杀、伤害、存活;我的队伍高亮置顶 | 任意比赛点开即看,数据来自本地库或免限流接口 |
| F5 | 新比赛自动检测 | 后台每 60s 轮询我的比赛列表,发现新比赛→入库→系统通知→点击直达详情页 | 比赛结束后 ≤3 分钟收到通知 |
| F6 | 车队面板 | 配置车队名单(≤10 人),批量拉取并排对比赛季数据 | 一次批量请求完成;支持增删改 |
| F7 | 本地库 | 比赛 JSON 永久入库(SQLite),战绩按 TTL 缓存 | 断网时历史数据可浏览 |

### P1 · 阶段二:游戏内(Overwolf)

| # | 功能 | 说明 | 验收标准 |
|---|---|---|---|
| F8 | 游戏检测与挂载 | 检测 PUBG 启动,自动启用游戏内能力;托盘常驻 | 从第二局起全程零手动操作 |
| F9 | 队友浮窗 | 进局后 GEP roster 拿到队友名 → 自动查询 → 浮窗展示每人 K/D、场均伤害、吃鸡率 | 进局(出生岛)后 ≤10s 显示;快捷键显隐 |
| F10 | 控制小浮窗 | 可拖动、半透明、常驻:图层开关(刷车点/密室/自定义)、地图呼出、锁定 | 拖动位置记忆;不遮挡准星区域 |
| F11 | 游戏内地图 V1 | 快捷键呼出自绘地图(Leaflet):标记图层齐全、可缩放拖动、**可当场右键添加标点** | 呼出 <1s;标点即时持久化 |
| F12 | 标记数据管理 | 桌面 App 内的地图编辑器:内置数据(密室/钥匙,手标)+ 用户自定义标点的增删改 | 与游戏内地图共用同一份数据 |
| F21 | 队伍存活追踪面板 | GEP roster 覆盖全场约 99 人(进局 airfield 阶段收齐),监听每人 `out` 状态翻转,浮窗实时显示各队剩余人数(队号 + 4 点阵,减员置灰,最近减员高亮);仅 squad 模式启用 | 减员后 ≤5s 面板更新;与游戏自带总存活数偏差 ≤2 |

### P2 · 阶段三:深度分析(Telemetry)

| # | 功能 | 说明 |
|---|---|---|
| F13 | Telemetry 管道 | 新比赛自动下载 telemetry(免限流),原始文件存盘,关键事件解析入库 |
| F14 | 死亡画像 | 死亡距离/武器/方位/阶段的汇总统计:「我总死于什么」 |
| F15 | 跳点复盘 | 各地图落点分布 × 存活时间/战绩热力图 |
| F16 | 谁杀了我 | 每次死亡:凶手历史战绩 + 那一枪的武器/距离,一键留档 |
| F17 | 刷车点自动聚类 | 聚合几百场 telemetry 的开局上车坐标,自动生成带热度的刷车点图层(反哺 F12) |

### P3 · 阶段四:完全体

| # | 功能 | 说明 |
|---|---|---|
| F18 | 屏幕识别对齐 V2 | 打开游戏 M 地图时,标记投影贴合到游戏地图正确位置;点游戏地图反向录入标点 |
| F19 | MCP Server | 把查询/分析能力暴露为 MCP 工具,在 Claude 里自然语言问战绩 |
| F20 | 车队周报 | 定时生成车队战报卡片(舔包王/本周之星),可推送到群 |

## 5. 桌面 App 页面清单

```
侧边栏导航
├── 仪表盘          我的概览卡片 + 近期比赛列表(F3)
├── 玩家查询        搜索框 + 战绩面板,模式切换 Tab(F2)
├── 比赛详情        (从列表进入)全员表格 + 我队高亮(F4)
├── 车队            成员管理 + 并排对比表(F6)
├── 地图标记        地图编辑器:选地图 → 图层 → 增删标点(F12)
├── 分析            死亡画像 / 跳点复盘 / 谁杀了我(阶段三)
└── 设置            API Key、绑定、轮询、快捷键、数据目录(F1)

游戏内(独立 overlay 窗口,共两个)
├── 地图标记浮窗(F11 → F18):打开游戏地图时展示标记点,快捷键 F8
└── 队伍存活浮窗(F21):仅存活点阵,快捷键 F9
    (队友战绩 F9 功能并入主窗口展示,不做独立浮窗 —— 2026-07-17 用户决策)
```

## 6. 非功能需求

- **性能**:缓存命中的页面 <300ms;游戏内浮窗渲染不影响游戏帧率(overlay 独立进程/窗口)
- **离线**:无网络时本地历史数据全部可用
- **数据**:单一 SQLite 文件 + telemetry 目录,放 userData,可整体备份;telemetry 原始文件默认只保留最近 300 场(可调),分析聚合结果永久保留
- **网络**:直连 `api.pubg.com`,失败时明确报错提示(含连通性排查提示)
- **平台**:仅 Windows 10/11(64 位),不支持 macOS/Linux

## 8. 风险

| 风险 | 应对 |
|---|---|
| Overwolf GEP 在游戏大版本后短暂失效 | 监控官方事件健康接口;降级为手动输入队友名 |
| PUBG 独占全屏模式下 overlay 不可见 | 引导使用无边框窗口化(主流设置);M4 实测验证 |
| 10 RPM 限流不够用 | 缓存 + 批量接口足以覆盖;不够再向官方申请提额 |
| 屏幕识别对齐工程量大 | 放最后(M7);V1 地图已可用,V2 失败不影响主体 |
| 地图底图素材版权 | 自用无碍;若开源,底图由使用者自行放入,仓库只含标记数据 |
| better-sqlite3 等原生模块与 Electron ABI | electron-builder 自动 rebuild,锁定版本组合 |
